**Captured**: 23:29 Europe/Dublin

## Overview

Address all issues identified in the Codex code review of commit `028ccaa`. The review identified critical cache invalidation bugs and several other issues that need fixing before the performance optimization can be considered complete.

## Goals

1. Fix critical cache key mismatch in invalidation
2. Add response cache invalidation after write operations
3. Add response caching for `get_tags` with `include_items=True`
4. Address race condition in `_fetch_search_cache`
5. Add cache size limits to prevent unbounded memory growth
6. Remove duplicate cache entries for tags

## Issues from Codex Review

### Critical
- **Cache key mismatch**: `invalidate_projects_cache()` clears `projects_cache`, but `get_projects` reads `projects_cache_include_items_*` and `projects_response_*`

### Major
1. No response cache invalidation after `add_todo`, `add_project`, `update_todo`, `update_project`
2. `get_tags` with `include_items=True` still makes per-tag DB calls (no response caching)
3. Race condition: `_fetch_search_cache` does read-modify-write without concurrency guard

### Minor
1. Unbounded session memory growth - no cache eviction
2. Duplicate caches - tags cache key includes `include_items` but `things.tags()` ignores it

## Key Decisions

### 1. Centralized Cache Key Management
Created a `CacheKeys` class with all cache key definitions to prevent key mismatches between cache storage and invalidation. Cache keys are now generated by methods like `CacheKeys.projects_response(include_items)`.

### 2. Individual Keys vs Shared Dict
Changed `_fetch_search_cache` from using a shared `search_cache` dict (which had race conditions during read-modify-write) to individual state keys via `CacheKeys.search_raw(key)`.

### 3. Cache Size Limits
Added `MAX_RESPONSE_CACHE_SIZE = 50` with simple LRU-style eviction. Cache keys are tracked in a list and oldest entries are evicted when the limit is exceeded.

### 4. Targeted Invalidation
Created separate invalidation functions for different cache types:
- `invalidate_projects_cache()` - clears project-related caches
- `invalidate_areas_cache()` - clears area-related caches
- `invalidate_todos_cache()` - clears search/recent/tagged caches
- `invalidate_tags_cache()` - clears tag-related caches
- `invalidate_all_caches()` - nuclear option, clears everything

## Changes Made

### Modified Files
- `src/things3_mcp/fast_server.py`:
  - Added `CacheKeys` class for centralized cache key management
  - Added `_track_cache_key()` for cache size tracking and eviction
  - Added `invalidate_*` functions for targeted cache invalidation
  - Updated all cache functions to use `CacheKeys` methods
  - Fixed `_fetch_search_cache` race condition (individual keys)
  - Added response caching to `_build_tags_response`
  - Added cache invalidation to `add_task`, `add_new_project`, `update_task`, `update_existing_project`
